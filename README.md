# Genxapi Ecosystem Demo

This repository is a working example project for `genxapi.dev`.

It shows how `genxapi` can sit at the center of a local multi-app workflow:

- NestJS services expose Swagger
- `genxapi` generates SDKs from those contracts
- Nx orchestrates the workspace locally
- a real consumer app uses the generated SDKs instead of hand-written API clients

At the current stage, the fully integrated consumer is `web-app`. The `backoffice-app` and `mobile-app` are present as future consumers, but they are not yet wired into the generated SDK flow.

## What This Repo Demonstrates

- A contract-first local workflow where service changes can flow into SDK generation and then into a consumer app
- The division of responsibility between `genxapi` and Nx
- A self-contained demo setup that works without publishing SDK packages first
- A simple cross-domain example with `users-service`, `payments-service`, and a web UI that consumes both

## Current Architecture

- `apps/users-service`
  - NestJS service on port `3001`
  - exposes Swagger UI and Swagger JSON
- `apps/payments-service`
  - NestJS service on port `3002`
  - exposes Swagger UI and Swagger JSON
- `sdk/users-sdk`
  - generated by `genxapi` from the users Swagger contract
- `sdk/payments-sdk`
  - generated by `genxapi` from the payments Swagger contract
- `apps/web-app`
  - React + Vite consumer
  - currently the only app integrated with the generated SDKs

## Why The Integration Is Shaped This Way

- `genxapi` owns SDK generation
  - each SDK config uses `beforeGenerate` to refresh Swagger before generating clients
- Nx owns workspace orchestration
  - it runs the services, SDK targets, and consumer targets together
- local file-based SDK dependencies keep the demo runnable on a laptop
  - no package publish step is required to try the system end to end

This keeps the example aligned with the point of `genxapi.dev`: the interesting part is not only that Nx can coordinate projects, but that `genxapi` can sit in the middle of the contract-to-consumer loop.

## Try It Locally

Recommended runtime:

- Node.js `20+`
  - `genxapi` currently emits engine warnings on Node `18`

Install dependencies:

```bash
npm install
```

If you also want to try the Expo starter separately:

```bash
npm install --prefix apps/mobile-app
```

Run the integrated local demo:

```bash
npm run serve:demo
```

Then open:

- Web app: `http://localhost:4200`
- Users Swagger UI: `http://localhost:3001/swagger`
- Payments Swagger UI: `http://localhost:3002/swagger`

## What Happens During `serve:demo`

1. Nx starts `users-service`, `payments-service`, and `web-app`.
2. Before `web-app` starts, its SDK dependencies trigger `genxapi generate`.
3. Each `genxapi` config runs `beforeGenerate`.
4. `beforeGenerate` refreshes the service Swagger JSON through the Nx export targets.
5. `genxapi` generates the SDKs.
6. `web-app` runs against those locally generated SDK packages.

## Useful Commands

Run the integrated demo:

```bash
npm run serve:demo
```

Refresh both SDKs manually:

```bash
npm run gen:sdks
```

Run only the web flow pieces:

```bash
nx serve users-service
nx serve payments-service
nx serve web-app
```

Inspect the workspace graph:

```bash
nx graph
```

## Swagger Endpoints

- Users service Swagger UI: `http://localhost:3001/swagger`
- Payments service Swagger UI: `http://localhost:3002/swagger`
- Users JSON: `http://localhost:3001/swagger-json`
- Payments JSON: `http://localhost:3002/swagger-json`

To pull Swagger JSON into `docs/swagger/` directly:

```bash
npm run swagger:pull
```

## Current Scope

What is integrated now:

- both backend services
- both generated SDKs
- `web-app` consuming the generated SDKs
- local Nx orchestration for the end-to-end demo

What is not integrated yet:

- `backoffice-app` consuming the SDKs
- `mobile-app` consuming the SDKs
- a production publish/distribution flow for the SDKs as part of the local demo story
